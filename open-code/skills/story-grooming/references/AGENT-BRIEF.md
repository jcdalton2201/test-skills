# AGENT-BRIEF.md: Template for Agent-Driven Development

This document provides the template and structure for "Agent Briefs"—instructions generated by the `story-grooming` skill for tasks with `ready` + `agent` labels.

An **Agent Brief** is a standalone document that tells an AI agent (or a developer) exactly what to build, how to test it, and how to get help if stuck.

---

## What Is an Agent Brief?

An Agent Brief is a complete, actionable specification for a single 2-day development task.

It includes:
- **Context**: What story this task belongs to, what problem it solves
- **What to build**: End-to-end description (testable, unambiguous)
- **Acceptance criteria**: Specific, testable checklist
- **Implementation steps**: Step-by-step guidance (not prescriptive, but helpful)
- **Testing expectations**: What tests to write, coverage thresholds
- **Definition of Done**: When is the task really complete?
- **Escalation path**: How to ask for help if stuck

## Acceptance Criteria

**Your task is complete when ALL of these checkmarks are checked:**

- [ ] **Database Table Created**
  - Table name: `challenge_qa`
  - Fields: `id` (UUID), `applicant_id` (UUID), `question_type` (enum), 
    `answer_encrypted` (TEXT), `created_at` (timestamp), `expires_at` (timestamp),
    `attempt_count` (int), `locked` (boolean)
  - All fields are documented in code

- [ ] **Constraints Applied**
  - NOT NULL: id, applicant_id, question_type, answer_encrypted, created_at
  - UNIQUE: (applicant_id, question_type) per expiry window
  - CHECK: attempt_count >= 0
  - Foreign key: applicant_id references applicants(id)

- [ ] **Indexes Created**
  - Index on (applicant_id, expires_at) for efficient cleanup queries
  - Index on (applicant_id, created_at) for audit trails

- [ ] **Encryption Implemented**
  - AES-256-GCM used for answer encryption
  - Initialization Vector (IV) stored with encrypted answer
  - Decryption roundtrip works correctly (encrypt → decrypt = original)
  - Encryption key is loaded from environment (not hardcoded)

- [ ] **Migration File Created**
  - File: `src/db/migrations/YYYYMMDD_create_challenge_qa.js`
  - Includes `up()` and `down()` functions
  - Migration can run and rollback successfully

- [ ] **Unit Tests Pass**
  - Test: Schema is created with correct fields
  - Test: Encryption/decryption roundtrip
  - Test: Constraints are enforced (NOT NULL, UNIQUE, CHECK)
  - Test: Indexes are created
  - Code coverage: ≥80% of new code

- [ ] **Integration Tests Pass**
  - Test: Migration runs successfully
  - Test: Migration can rollback
  - Test: No errors in staging database

- [ ] **Documentation Complete**
  - Schema design documented (fields, why chosen, constraints rationale)
  - Encryption approach documented (algorithm, key rotation policy)
  - Migration documented (how to deploy, rollback steps)

## Implementation Approach

### Step 1: Understand the Existing Codebase

Before you start coding:

1. **Review existing migrations**: Look at `src/db/migrations/` to understand the pattern
2. **Review existing models**: Look at `src/models/` to understand Sequelize setup
3. **Review encryption service**: Check `src/services/encryption/` if it exists
4. **Ask**: If anything is unclear, ask the team before spending time

**Time estimate**: 30 minutes

---

### Step 2: Create Migration File

Create migration file: `src/db/migrations/20260511_create_challenge_qa.js`

The migration should:
- Define table with all fields (see Acceptance Criteria)
- Add constraints and indexes
- Include rollback (`down()` function)

**Reference**: Look at existing migrations in the same directory for the pattern.

**Time estimate**: 1-2 hours

---

### Step 3: Create Sequelize Model

Create model file: `src/models/ChallengeQA.ts` (or `.js` if project is JS-only)

The model should:
- Define schema matching the migration
- Include validation (NOT NULL, constraints)
- Include encryption/decryption methods
- Include any utility methods (e.g., `getValidAnswer()`, `isExpired()`)

**Time estimate**: 2-3 hours

---

### Step 4: Create Encryption Service

If encryption service doesn't exist, create: `src/services/encryption/index.ts`

The service should:
- Export `encrypt(text, key)` function (returns encrypted blob with IV)
- Export `decrypt(blob, key)` function (returns original text)
- Use AES-256-GCM from Node.js crypto module
- Store IV with encrypted blob (JSON or concatenated)

**Time estimate**: 2-3 hours

**Or**: If encryption service already exists, integrate with it (add tests).

---

### Step 5: Write Unit Tests

Create test file: `src/db/__tests__/ChallengeQA.test.ts`

Tests should cover:
- Schema creation and fields
- Encryption/decryption roundtrip
- Constraint validation (NOT NULL, UNIQUE, CHECK)
- Index creation
- Model methods

**Time estimate**: 3-4 hours

---

### Step 6: Write Integration Tests

Create test file: `src/db/__tests__/migration.integration.test.ts`

Tests should cover:
- Migration runs without errors
- Migration rollback works
- Table and indexes exist in database
- Constraints are enforced at database level

**Run against staging database** before merge.

**Time estimate**: 2-3 hours

---

### Step 7: Document

Create/update: `docs/schema/challenge_qa.md`

Documentation should include:
- Table purpose and business context
- Schema design rationale (why these fields)
- Encryption approach (algorithm, key management)
- Migration instructions (how to deploy, how to rollback)
- Example queries (how to insert, retrieve, validate answers)

**Time estimate**: 1-2 hours

---

### Step 8: Code Review & Merge

1. Commit all changes: migration, model, tests, encryption service, docs
2. Push branch and create PR
3. Link PR to Jira task: "Fixes CARD-101-A-1"
4. Wait for code review
5. Address feedback
6. Merge to main branch

**Time estimate**: 1-2 hours (waiting for review not included)

## Testing Expectations

### Unit Tests
- **Coverage**: ≥80% of new code
- **Tools**: Jest (already in project)
- **Run**: `npm test` or `jest`
- **What to test**:
  - Each model method
  - Each encryption/decryption path
  - Edge cases (NULL inputs, very long answers, etc.)

### Integration Tests
- **Coverage**: Database interactions
- **Tools**: Jest + test database (use Docker or test instance)
- **Run**: `npm test:integration`
- **What to test**:
  - Migration runs successfully
  - Rollback works
  - Constraints enforced by database (not just ORM)
  - Indexes exist

### Manual Testing (Optional)
```bash
# Run migration locally
npm run db:migrate

# Verify table exists
psql -c "\d challenge_qa"

# Insert a test record
INSERT INTO challenge_qa (applicant_id, question_type, answer_encrypted, expires_at)
VALUES (uuid(), 'SSN_LAST_4', 'encrypted_answer', now() + interval '24 hours');

# Verify index
SELECT * FROM pg_indexes WHERE tablename = 'challenge_qa';
```

## If You Get Stuck

### Stuck on: "Where should code go?"
- Check existing directories for similar files
- Ask team: "Which directory for X?"
- Default: `src/models/` for models, `src/services/` for services, `src/db/` for migrations

### Stuck on: "How do I write a Sequelize model?"
- Reference: Sequelize docs (https://sequelize.org/)
- Reference: Existing models in `src/models/`
- Ask team for code review on draft

### Stuck on: "Encryption is too complex"
- Ask: "Does encryption service already exist in the codebase?"
- If yes, use it (don't rewrite)
- If no, ask security team for guidance on key management
- Default: Use Node.js `crypto` module with AES-256-GCM

### Stuck on: "Tests are failing"
- Debug locally: Run test with `--verbose` flag
- Check error message: What's actually failing?
- If database-related: Did migration run? Is test database set up?
- Ask: "Can someone pair with me to debug?"

### Stuck on: "Task is bigger than 2 days"
- Flag it: "This is taking longer than 2 days"
- Ask: "Should this task be split, or is the 2-day estimate wrong for our team?"
- Document: How long it actually took (learn for next estimate)
- Don't force it into 2 days; communicate early

## Escalation Path

**If you're blocked or unsure:**

1. **First**: Check documentation (schema docs, existing models, tests)
2. **Second**: Ask team in Slack or comment on Jira task
3. **Third**: Escalate to tech lead: "Need guidance on X, blocking my progress"
4. **Last resort**: Re-mark task as `needs-info` if the requirement is genuinely ambiguous

**Expected response time**: 4 hours during business hours
